# 健康记录本 Zero-Friction 设计

日期：2026-04-13

## 目标

把“血压心率记录”小程序从“先创建家庭组/档案”调整为“打开即记录”的低门槛模型。

核心目标：

- 用户第一次打开即可录入血压、心率。
- 首次保存时后台静默创建默认记录本，不打断记录流程。
- 档案信息、家人协作、多记录本管理都作为后续增强能力出现。
- 保留现有数据页和家庭页的视觉骨架，做最小 UI 改动。

## 设计结论

前台主概念改为“健康记录本”。

```text
1 个记录本 = 1 组数据 = 1 套档案 = 1 套成员权限 = 1 套邀请链接/邀请码
```

不采用“1 个家庭组包含多个记录本，加入后可看全部记录本”的模型。

原因：

- 健康数据需要最小权限授权。
- 邀请哥哥照顾爸爸，不应默认让哥哥看到“我的记录”或“妈妈的记录”。
- 对用户来说，“邀请你查看爸爸的记录”比“加入家庭组”更容易理解。
- 家庭 tab 可以继续存在，但它展示的是当前记录本的照护协作状态。

## 页面结构

底部 tab 保持两个：

```text
数据
家庭
```

不新增设置 tab。

### 数据 tab

数据 tab 是主体验，负责：

- 录入血压/心率。
- 显示最近一次记录。
- 显示 7 / 30 / 90 天趋势。
- 下载血压/心率图表。
- 进入全部记录。

第一次打开时，数据 tab 直接显示录入区，不要求创建档案：

```text
血压心率记录

高压
低压
心率

保存记录

查看家人的记录？输入邀请码
```

首次点击“保存记录”时：

- 后台静默创建默认记录本 `我的记录`。
- 当前记录写入 `我的记录`。
- 页面进入正常数据分析态。

### 家庭 tab

家庭 tab 展示当前记录本的协作状态，负责：

- 当前记录本档案卡。
- 照护成员列表。
- 邀请家人。
- 生成就诊报告。
- 药物管理入口。
- 设置入口。

家庭页标题仍然可以显示 `家庭`，但文案避免使用“家庭组”。优先使用：

- `照护成员`
- `邀请家人查看这个记录`
- `当前记录本成员`
- `爸爸的记录`

## 全局 Switcher

顶部标题栏支持记录本切换。

默认显示：

```text
血压心率记录
```

当存在多个记录本或用户点击当前标题时，作为 Switcher 展开：

```text
我的记录
爸爸的记录
妈妈的记录
+ 新增记录本
输入邀请码
```

切换记录本后同步影响：

- 数据 tab 的最近记录、图表、全部记录、添加记录权限。
- 家庭 tab 的档案、照护成员、邀请、权限设置。
- 报告导出对象。

## 三个初始场景

### 场景 A：自用先锋

用户第一次打开，给自己记录。

流程：

```text
打开小程序
-> 数据 tab 直接录入
-> 保存第一条记录
-> 静默创建“我的记录”
-> 正常显示趋势和下载图表
-> 家庭 tab 可选完善档案
-> 可选邀请配偶/家人查看
```

关键规则：

- 不弹出“创建档案”页面。
- 不强制填写姓名、出生年、用药、联系人。
- 档案卡显示 `我的记录`，提示“档案信息可稍后补充”。

### 场景 B：照护核心

用户第一次打开，给长辈记录。

流程：

```text
打开小程序
-> 数据 tab 直接录入
-> 保存第一条记录
-> 静默创建“我的记录”
-> 用户通过 Switcher 或家庭页把记录本重命名为“爸爸的记录”
-> 可选完善出生年、当前用药、紧急联系人
-> 导出就诊报告
```

关键规则：

- 即使用户最终是给爸爸记录，也不在首次录入前强制选择“给谁记录”。
- 重命名和档案完善发生在记录之后。
- 当前用药、紧急联系人只用于报告增强，始终选填。
- 如果用户后来收到另一个“爸爸的记录”邀请，不自动合并当前默认记录本，必须让用户选择保留、复制或移动。

### 场景 C：纯观察员

用户通过家人邀请进入，只看别人数据。

流程：

```text
点击微信邀请卡片
-> 进入邀请确认页
-> 接受邀请
-> Switcher 出现“爸爸的记录”
-> 数据 tab 显示爸爸数据
-> 添加按钮置灰，说明“仅可查看”
-> 用户可申请协助录入权限
-> 用户也可一键创建“我的记录”
```

关键规则：

- 默认加入只给查看权限。
- `canWrite=false` 时添加按钮不可用，但必须说明原因。
- “为自己创建记录”是轻入口，不打断观察爸爸数据的主任务。

## 邀请模型

邀请必须针对当前记录本，不针对用户创建的所有记录本。

主路径：微信分享卡片。

```text
Ella 邀请你
共同查看爸爸的健康记录

默认权限：可查看
协助录入：管理员开启后可用
```

分享 path 需要携带：

```text
inviteToken
recordBookId
```

兜底路径：手动邀请码。

MVP 调整：手动邀请码入口暂不在前端展示。当前只上线微信分享卡片路径；后端 `inviteCode` 兼容可以保留，等确实需要兜底路径时再做邀请码长度、hash 存储、限流和锁定策略。

入口文案：

```text
查看家人的记录？输入邀请码
```

邀请码规则：

- 与单个记录本绑定。
- 默认 24 小时有效；如果业务坚持 48 小时，邀请码长度必须增加。
- 手动邀请码至少 8 位，使用不易混淆的字符集。
- 服务端只保存邀请码 hash，不保存明文码。
- 服务端按 openid、设备和 IP 维度限流。
- 连续失败后短时间锁定验证入口。
- 过期后提示让邀请人重新发送。
- 手动输入页说明“如果收到微信邀请卡片，直接点击卡片更方便”。

邀请管理展示：

```text
备用邀请码 A7K92Q
有效期：剩余 23 小时
已发送给：大哥
默认权限：可查看
```

## 权限模型

面向用户的权限角色：

```text
观察员 viewer: 查看记录
协作者 contributor: 新增记录，编辑自己新增的记录
管理员 admin: 管理档案、设置、邀请、权限、全部记录
```

默认加入：

```text
role = viewer
```

技术实现可以继续保存 `canView`、`canWrite`、`canEdit`，但 UI 不直接暴露这些字段。V1 中：

- viewer：`canView=true`，`canWrite=false`，`canEdit=false`。
- contributor：`canView=true`，`canWrite=true`，可编辑自己创建的记录。
- admin：可管理当前记录本的全部记录、档案、设置、成员和邀请。

权限冲突处理：

- 没有录入权限时，添加按钮置灰或显示不可用态。
- 点击不可用按钮时显示：

```text
你目前只有查看权限，无法添加记录。请联系管理员开启协助录入。
```

- 没有编辑权限时，修改/删除操作显示：

```text
你目前只有查看权限，无法修改记录。请联系管理员开启编辑权限。
```

复杂操作可给弹窗：

```text
无法修改这条记录
你现在是观察员，只能查看爸爸的健康记录。
需要修改或删除记录时，请联系管理员开启编辑权限。

我知道了
联系管理员
```

## 记录本合并与迁移

打开即记录会带来一个风险：用户可能先在默认 `我的记录` 中为爸爸记录，之后又加入真正的 `爸爸的记录`。

系统不能自动合并两本记录，避免误把自己的数据并入长辈记录。

当用户接受邀请或切换到新记录本时，如果检测到本地已有默认记录本且存在记录，展示选择：

```text
你已经有一批记录
是否要把它们用于“爸爸的记录”？

保留为我的记录
复制到爸爸的记录
移动到爸爸的记录
```

V1 最保守策略：

- 默认选择“保留为我的记录”。
- 允许后续从全部记录页批量选择并移动到另一个记录本。
- 移动/复制时保留审计字段：

```text
sourceRecordBookId
migratedFromRecordId
migratedAt
migratedBy
```

如果用户选择“移动”，原记录本中不再展示这些记录，但后台保留迁移日志。

## 离线与并发

新增记录需要支持弱网场景。

V1 策略：

- 新增记录可离线暂存。
- 离线首次保存时，本地创建 `localRecordBookId` 和 `localRecordId`。
- 网络恢复后先创建远端 recordBook，再上传本地 records。
- 如果远端创建失败，记录保留在本地待同步队列，不丢弃用户输入。
- 图表本地先基于待同步记录展示，并标记“待同步”。

编辑和删除策略：

- V1 要求在线编辑和删除，降低冲突复杂度。
- 每条记录保存 `version` 和 `updatedAt`。
- 如果后续支持离线编辑，服务端需检查 version；版本不一致时返回冲突。
- 两个成员在相近时间新增记录不视为冲突，按两条记录保留。
- 两个成员修改同一条记录时，提示：

```text
这条记录已被其他成员更新。
请刷新后再编辑。
```

## 档案与隐私

档案不是使用门槛。

必需：

- 记录本名称，可系统默认 `我的记录`。

选填：

- 昵称/称呼。
- 出生年。
- 当前用药。
- 紧急联系人姓名。
- 紧急联系人电话。

信任文案：

```text
档案信息仅用于记录本展示和报告生成。
您可以填写昵称或简称，除记录需要外，不必填写真实身份信息。
```

不承诺尚未实现的“加密”能力。

合规边界：

- 医疗健康信息属于敏感个人信息，必须在隐私政策和页面提示中说明处理目的、方式、范围和保存期限。
- 用户需要能删除记录、删除记录本、移除成员。
- 成员被移除后，不可再访问对应记录本。
- 用户撤回授权或删除记录本后，后端需要停止对该记录本数据的共享访问。
- 记录本删除前应二次确认，并说明删除后会影响图表和报告。

隐私政策需要覆盖：

```text
收集哪些数据：血压、心率、测量时间、记录本名称、选填档案、成员权限。
用于什么：记录展示、趋势统计、图表下载、报告生成、受邀成员协作查看。
谁能查看：用户本人和被授权加入当前记录本的成员。
保存多久：用户删除前保存，或按后续隐私政策声明的期限保存。
如何删除：在设置中删除记录、记录本或成员授权。
```

## 图表与报告

下载图表：

- 不需要档案。
- 直接导出时间段内血压/心率图。

生成就诊报告：

- 可以匿名导出。
- 档案完善后，报告可包含姓名/称呼、出生年、当前用药、紧急联系人。
- 导出时提供隐私选项：

```text
隐藏真实姓名/电话
```

如果档案缺失：

```text
可先导出匿名报告。
完善档案后，报告会包含姓名、出生年、用药等信息。
```

## 最小 UI 改动策略

保留现有视觉骨架：

- 数据页蓝色顶部主卡。
- 添加记录主按钮。
- 7 / 30 / 90 天切换。
- 血压/心率图表卡。
- 家庭页档案卡。
- 家庭页成员横向列表。
- 家庭页功能卡：生成就诊报告、药物管理。
- 家庭页右上设置入口。

新增小入口：

- 数据页轻链接：`查看家人的记录？输入邀请码`。
- 顶部标题 Switcher。
- 家庭页邀请入口。
- 观察员视角的 `为自己创建记录`。
- 报告导出的 `隐藏真实姓名/电话`。

## 实施顺序建议

1. 数据页支持无家庭/无档案直接录入，首次保存静默创建默认记录本。
2. 引入当前记录本状态和 Switcher。
3. 记录查询、保存、删除改为按 recordBookId 过滤。
4. 邀请链接和邀请码改为绑定 recordBookId，并加入限流、过期、hash 存储。
5. 家庭页改为展示当前记录本的成员和档案。
6. 权限 UI 改为 viewer / contributor / admin 角色。
7. 补默认记录本迁移/复制/保留策略。
8. 补离线新增记录待同步队列。
9. 报告导出加匿名/隐藏真实姓名电话选项。
10. 补隐私政策入口、删除记录本、移除成员能力。
